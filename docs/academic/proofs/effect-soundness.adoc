= Algebraic Effects: Formal Semantics and Soundness

*Document Version*: 1.0 *Last Updated*: 2024 *Status*: Theoretical
framework complete; implementation verification pending
`++[++IMPL-DEP: effect-checker, effect-inference++]++`

== Abstract

This document presents the formal semantics of AffineScriptвҖҷs algebraic
effect system. We define the syntax, typing rules, and operational
semantics for effects and handlers, proving type safety and effect
safety: well-typed programs only perform effects that are handled.

== 1. Introduction

Algebraic effects and handlers provide a structured approach to
computational effects, separating effect signatures from their
implementations. AffineScriptвҖҷs effect system features:

[arabic]
. *User-defined effects*: Custom effect declarations
. *Row-polymorphic effects*: `ОөвӮҒ ++|++ ОөвӮӮ ++|++ ..ПҒ`
. *One-shot and multi-shot handlers*: Controlled use of continuations
. *Effect polymorphism*: Abstracting over effect rows
. *Effect inference*: Automatic effect tracking

== 2. Syntax

=== 2.1 Effect Declarations

....

}

=== 2.2 Effect Types

....
Оө ::=
    | В·                           -- Empty effect (pure)
    | Op                          -- Single effect operation
    | E                           -- Named effect
    | ОөвӮҒ | ОөвӮӮ                     -- Effect union

....

=== 2.3 Effectful Function Types

....

=== 2.4 Effect Operations and Handlers

....

....

=== 3.2 Effect Row Operations

*Row Equivalence*: Effect rows are equivalent up to: - Commutativity:
`ОөвӮҒ ++|++ ОөвӮӮ вүЎ ОөвӮӮ ++|++ ОөвӮҒ` - Associativity:
`(ОөвӮҒ ++|++ ОөвӮӮ) ++|++ ОөвӮғ вүЎ ОөвӮҒ ++|++ (ОөвӮӮ ++|++ ОөвӮғ)` - Identity:
`Оө ++|++ В· вүЎ Оө` - Idempotence: `E ++|++ E вүЎ E`

*Row Subtraction*: `Оө ++\++ E` removes effect E from row Оө

....
    В· \ E = В·
    E \ E = В·
    E' \ E = E'   (when E вү  E')
    (ОөвӮҒ | ОөвӮӮ) \ E = (ОөвӮҒ \ E) | (ОөвӮӮ \ E)
    ПҒ \ E = ПҒ     (row variable, handled later)
....

=== 3.3 Typing Effectful Expressions

The judgment `О“ вҠў e : П„ ! Оө` means "`in context О“, expression e has type
П„ and may perform effects Оө`".

*Pure expressions*:

....
    О“ вҠў e : П„
    в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ
    О“ вҠў e : П„ ! В·
....

*Effect Subsumption*:

....
    О“ вҠў e : П„ ! ОөвӮҒ    ОөвӮҒ вҠҶ ОөвӮӮ
    в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ
    О“ вҠў e : П„ ! ОөвӮӮ
....

=== 3.4 Effect Operation Typing

*Perform*

....
    op : П„ вҶ’ Пғ вҲҲ E
    О“ вҠў e : П„ ! Оө
    в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ
    О“ вҠў perform op(e) : Пғ ! (E | Оө)
....

=== 3.5 Handler Typing

*Handle*

....
    О“ вҠў e : П„ ! (E | Оө)
    О“ вҠў h handles E : П„ вҮ’ Пғ ! Оө'
    в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ
    О“ вҠў handle e with h : Пғ ! (Оө | Оө')
....

*Handler Clause Typing*

....
    О“, x:П„_ret вҠў e_ret : Пғ ! Оө'
    вҲҖ op вҲҲ E. О“, x:П„_op, k:(Пғ_op вҶ’{Оө | Оө'} Пғ) вҠў e_op : Пғ ! Оө'
    в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ
    О“ вҠў { return x вҶ’ e_ret, op(x,k) вҶ’ e_op, ... } handles E : П„_ret вҮ’ Пғ ! Оө'
....

Where: - `П„++_++ret` is the return type of the handled computation - For
each `op : П„++_++op вҶ’ Пғ++_++op` in E - `k` is the continuation, typed as
`Пғ++_++op вҶ’++{++Оө ++|++ Оө'} Пғ`

*Resume*

    О“, k:(Пғ вҶ’{Оө} П„_result) вҠў resume(e) : П„_result ! Оө
    в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ

== 4. Dynamic Semantics

=== 4.1 Values and Evaluation Contexts

*Values*:

....
v ::= ... | handler h
....

*Pure Evaluation Contexts*:

....
E_p ::= в–Ў | E_p e | v E_p | ...
....

*Effectful Evaluation Contexts*:

....
E_eff ::= E_p | handle E_eff with h
....

=== 4.2 Reduction Rules

*Handler Introduction*:

....
    в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ
    handle v with h вҹ¶ e_ret[v/x]
    (where h = { return x вҶ’ e_ret, ... })
....

*Effect Forwarding* (effect not handled):

....
    op вҲү E
    в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ
    handle E_p[perform op(v)] with h вҹ¶
        perform op(v) >>= (О»y. handle E_p[y] with h)
....

*Effect Handling*:

....
    op : П„ вҶ’ Пғ вҲҲ E
    h = { ..., op(x, k) вҶ’ e_op, ... }
    в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ
    handle E_p[perform op(v)] with h вҹ¶
        e_op[v/x, (О»y. handle E_p[y] with h)/k]
....

The key insight: the continuation `k` captures the context `E++_++p`,
allowing the handler to resume computation.

=== 4.3 One-Shot vs Multi-Shot Continuations

*One-shot* (linear k):

....
    k used exactly once in e_op
    Continuation can be efficiently implemented as a stack
....

*Multi-shot* (unrestricted k):

....
    k may be used zero, one, or multiple times
    Requires copying/delimited continuation implementation
....

AffineScript tracks this via quantity annotations:

....
    op(x, 1 k) вҶ’ e_op     -- k is linear (one-shot)
    op(x, Пү k) вҶ’ e_op     -- k is unrestricted (multi-shot)
....

== 5. Effect Safety

=== 5.1 Main Theorem

*Theorem 5.1 (Effect Safety)*: If `В· вҠў e : П„ ! В·` (closed, pure), then
evaluation of e does not get stuck on an unhandled effect.

*Proof*: We prove a stronger statement by showing that effects are
always contained within handlers.

Define "`effect-safe configuration`" inductively: 1. Values are
effect-safe 2. `handle e with h` is effect-safe if e may only perform
effects in dom(h) вҲӘ Оө where Оө are effects propagated outward

By progress and preservation for effects. вҲҺ

=== 5.2 Effect Preservation

*Theorem 5.2 (Effect Preservation)*: If `О“ вҠў e : П„ ! Оө` and `e вҹ¶ e'`,
then `О“ вҠў e' : П„ ! Оө'` where `Оө' вҠҶ Оө`.

*Proof*: By induction on the reduction.

_Case Handler-Return_: `handle v with h вҹ¶ e++_++ret++[++v/x++]++` The
handler removes the handled effect, so `Оө' = Оө ++\++ E вҠҶ Оө`. вң“

_Case Effect-Handle_: The handled effect is captured; remaining effects
are preserved. вң“

вҲҺ

=== 5.3 Effect Progress

*Theorem 5.3 (Effect Progress)*: If `В· вҠў e : П„ ! Оө` then either: 1. e is
a value, or 2. e can reduce, or 3. e = `E++[++perform op(v)++]++` where
`op вҲҲ Оө`

Case 3 represents a "`stuck`" effect, but this is only possible if Оө вү 
В·.

*Corollary*: If `В· вҠў e : П„ ! В·`, then e does not get stuck on effects.

== 7. Effect Inference

=== 7.1 Algorithm

Effect inference follows bidirectional type checking:

[source,ocaml]
----
(* Infer effects of an expression *)
val infer_effects : ctx -> expr -> (typ * effect) result

(* Check effects against expected *)
val check_effects : ctx -> expr -> typ -> effect -> unit result
----

Key cases:

[source,ocaml]
----
let rec infer_effects ctx = function
  | Perform (op, arg) ->
      let (eff, param_ty, ret_ty) = lookup_operation op in
      let _ = check ctx arg param_ty in
      Ok (ret_ty, eff)

  | Handle (body, handler) ->
      let (body_ty, body_eff) = infer_effects ctx body in
      let handled_eff = effects_of_handler handler in
      let remaining_eff = subtract body_eff handled_eff in
      let result_ty = return_type_of_handler handler body_ty in
      Ok (result_ty, remaining_eff)

  | App (f, arg) ->
      let (f_ty, f_eff) = infer_effects ctx f in
      match f_ty with
      | TyArrow (param_ty, ret_ty, fn_eff) ->
          let arg_eff = check_effects ctx arg param_ty in
          Ok (ret_ty, union [f_eff; arg_eff; fn_eff])
      | _ -> Error "not a function"
----

=== 7.2 Effect Constraint Solving

Generate and solve constraints:

....
ОөвӮҒ вҠҶ ОөвӮӮ                     -- Subsumption
ОөвӮҒ | ОөвӮӮ = ОөвӮғ                -- Composition
Оө \ E = Оө'                  -- Subtraction
ПҒ = Оө                       -- Row variable instantiation
....

=== 9.1 IO Effect

----
effect IO {

    read_file : String вҶ’ String

=== 9.2 State Effect

[source,affinescript]
----
effect State[S] {
    get : () вҶ’ S
    put : S вҶ’ ()
}

-- Derived operations
fn modify[S](f: S вҶ’ S) -> () / State[S] {
    put(f(get()))
}

-- Standard handler: run with initial state
fn run_state[S, A](init: S, comp: () вҶ’{State[S]} A) -> (A, S) {
    handle comp() with {
        return x вҶ’ (x, init),
        get(_, k) вҶ’ resume(k, init),
        put(s, k) вҶ’ run_state(s, О»(). resume(k, ()))
    }
}
----

=== 9.3 Exception Effect

[source,affinescript]
----

    raise : E вҶ’ вҠҘ

fn catch[E, A](comp: () вҶ’{Exn[E]} A, handler: E вҶ’ A) -> A {
    handle comp() with {
        return x вҶ’ x,

}

== 10. Categorical Semantics

=== 10.1 Free Monad Interpretation

Effects correspond to free monads:

....
Free E A = Return A | Op (ОЈ opвҲҲE. П„_op Г— (Пғ_op вҶ’ Free E A))
....

*Theorem 10.1*: The effect system is sound with respect to the free
monad semantics.

=== 10.2 Handler as Algebra

A handler for effect E is an E-algebra:

....
alg : E (Пғ вҶ’ A) вҶ’ A
....

The handle construct applies the algebra to eliminate the effect.

=== 10.3 Relationship to Monads

*Theorem 10.2*: For any effect E, running under a handler is equivalent
to interpreting in the corresponding monad.

== 11. Examples
